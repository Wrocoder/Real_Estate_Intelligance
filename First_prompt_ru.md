# Промпт для разработки SaaS-платформы аналитики недвижимости

Разработай полную концепцию, техническое задание, архитектуру, MVP-план, стратегию монетизации и оценку коммерческого потенциала для SaaS-платформы аналитики недвижимости в Польше с возможностью масштабирования на другие страны.

Рабочее название проекта: **Domarion Analytics** или **Real Estate Intelligence Poland**.

Документ должен быть написан на русском языке, четко, структурированно, практически и без воды. Это должен быть не рекламный текст, а реальный план, по которому можно начинать разработку, проверку спроса и поиск первых клиентов.

---

## 1. Главная идея продукта

Нужно создать SaaS-платформу для аналитики недвижимости в Польше, которая помогает пользователям принимать более умные решения при покупке, продаже, аренде и инвестировании в недвижимость.

Платформа должна быть полезна как обычному человеку, который покупает первую квартиру, так и профессиональному риелтору, инвестору, агентству недвижимости, ипотечному брокеру или аналитической компании.

Главная ценность продукта:

**Показывать не только текущие объявления, а объяснять, почему объект стоит именно столько, какие у него риски, насколько цена справедлива и что может повлиять на его стоимость в будущем.**

Пример:

Пользователь хочет купить 3-комнатную квартиру недалеко от Вроцлава за 300 000 zł. Платформа должна показать:

- текущую цену;
- цену за м²;
- историю изменения цены;
- похожие объекты рядом;
- среднюю и медианную цену за м² в районе;
- сколько похожих квартир сейчас продается;
- растет или падает количество объявлений;
- можно ли торговаться;
- сколько дней объект на рынке;
- что есть рядом сейчас;
- что планируется построить рядом в ближайшие 3–10 лет;
- будет ли новая школа, трамвайная линия, автобусное сообщение, дорога, парк, торговый центр;
- есть ли негативные факторы: завод, склады, шумная дорога, железная дорога, риск затопления, промзона;
- насколько объект подходит для жизни;
- насколько объект подходит для инвестиций;
- насколько объект подходит для аренды;
- может ли район вырасти в цене.

## 2. Проблема, которую решает продукт

Покупка недвижимости — одно из самых дорогих решений в жизни человека, но большинство покупателей принимает его на основе неполной информации.

Проблемы покупателей:

- не знают, завышена ли цена;
- не понимают, хороший ли район;
- не знают историю цены;
- не умеют сравнивать похожие объекты;
- не знают, как проверить инфраструктуру и планы развития района;
- боятся купить объект с плохой ликвидностью;
- не понимают, можно ли торговаться;
- не видят скрытые риски;
- не знают полной стоимости покупки с ипотекой, налогами, нотариусом и ремонтом.

Проблемы риелторов:

- сложно аргументировать цену клиенту;
- сложно быстро готовить аналитические отчеты;
- сложно сравнивать районы;
- нет удобного инструмента для анализа истории цены;
- нет автоматических отчетов по объектам;
- нет удобной карты будущей инфраструктуры;
- приходится вручную собирать данные из разных источников.

Проблемы инвесторов:

- сложно находить объекты ниже рынка;
- сложно оценить потенциал района;
- сложно понять будущую ликвидность;
- сложно сравнить арендную доходность;
- сложно быстро отсеивать плохие варианты.

## 3. Целевая аудитория

### 3.1. Покупатель первой квартиры

Пользователь, который хочет купить квартиру для себя или семьи.

Ему нужны простые ответы:

- нормальная ли цена;
- хорошая ли локация;
- есть ли риск переплаты;
- будет ли район развиваться;
- есть ли рядом транспорт, школы, магазины, парки;
- какие есть минусы;
- можно ли торговаться;
- какую ипотеку он может себе позволить;
- какие дополнительные расходы будут при покупке;
- какие документы проверить.

Интерфейс для него должен быть простым:

- “хороший вариант”;
- “переоценено”;
- “можно торговаться”;
- “есть риски”;
- “интересно для инвестиций”;
- “лучше сравнить с другими объектами”.

### 3.2. Риелтор

Пользователь, которому нужна профессиональная аналитика для клиентов.

Ему нужны:

- отчеты по объектам;
- сравнение с рынком;
- история цены;
- аргументы для торга;
- аналитика района;
- white-label PDF-отчеты;
- подборки объектов;
- экспорт данных;
- презентации для клиентов.

### 3.3. Инвестор

Пользователь, который ищет объекты с потенциалом роста или доходности.

Ему нужны:

- объекты ниже рынка;
- районы с потенциалом роста;
- арендная доходность;
- ликвидность;
- инфраструктурные планы;
- прогноз цены;
- риск-скоринг;
- уведомления о “hidden gems”.

### 3.4. Агентства недвижимости

Им нужны:

- командные аккаунты;
- отчеты под своим брендом;
- аналитика рынка;
- мониторинг конкурентов;
- CRM-light;
- API или экспорт данных;
- инструменты для презентации объектов клиентам.

### 3.5. Банки, брокеры, девелоперы, аналитические компании

В перспективе им можно продавать:

- API;
- market reports;
- lead generation;
- scoring объектов;
- районную аналитику;
- аналитику спроса и предложения.

## 4. География MVP

MVP нужно делать узко, чтобы не превратить проект в вечную стройку, как половина дорог в Восточной Европе.

Стартовый регион:

- Польша;
- Dolnośląskie;
- Wrocław;
- города и населенные пункты вокруг Вроцлава;
- радиусы: 5, 10, 20, 30 км от Вроцлава.

Тип недвижимости для MVP:

- квартиры на продажу;
- первичный и вторичный рынок.

После проверки MVP масштабировать:

- вся Польша;
- аренда;
- дома;
- земельные участки;
- коммерческая недвижимость;
- Чехия;
- Германия;
- Испания;
- другие страны ЕС.

Нужно предложить легальную и устойчивую стратегию получения данных.

Нельзя строить весь бизнес только на агрессивном скрапинге одного сайта. Это риск блокировок, юридических проблем и технической нестабильности.

### 5.1. Источники объявлений

Потенциальные источники:

- Otodom;
- OLX Nieruchomości;
- Morizon;
- Gratka;
- Nieruchomosci-online;
- сайты девелоперов;
- сайты агентств недвижимости;
- прямые партнерские интеграции с агентствами;
- ручная загрузка объектов риелторами;
- paid data providers;
- легальные API, если доступны.

Нужно продумать:

- соблюдение Terms of Service;
- robots.txt;
- rate limiting;
- хранение только нужных данных;
- хранение ссылки на оригинальное объявление;
- отказ от копирования фото без прав;
- отказ от публикации контактных данных без основания;
- дедупликацию объявлений;
- удаление данных по запросу;
- юридическую модель использования данных.

### 5.2. Официальные и открытые данные

Использовать:

- Geoportal / GUGiK;
- Rejestr Cen Nieruchomości;
- GUS / Bank Danych Lokalnych;

- местные geoportale городов;
- geoportal Wrocław;
- MPZP;
- Studium;
- данные о школах, детских садах, больницах;
- данные об остановках и маршрутах транспорта;
- муниципальные инвестиции;
- планы дорог;
- транспортные планы;
- OpenStreetMap;
- GTFS, если доступен;
- пресс-релизы городов, гмин, воеводств;
- новости девелоперов;
- новости министерств и банков.

- этаж;
- количество этажей в доме;
- год постройки;
- рынок первичный / вторичный;
- тип здания;
- состояние ремонта;
- балкон;
- терраса;
- сад;
- лифт;
- парковка;
- отопление;
- расстояние до центра;
- расстояние до остановки;
- расстояние до школы;
- расстояние до крупной дороги;
- расстояние до промзоны;
- Investment Score;
- Risk Score;
- Liquidity Score;
- Rent Potential Score.

### 6.2. Карта объектов

Карта должна показывать:

- объявления;
- цену за м²;
- тепловую карту цен;
- районы;
- гмины;
- транспорт;
- школы;
- детские сады;
- магазины;
- больницы;
- парки;
- промышленные зоны;
- строящиеся объекты;
- будущие дороги;
- будущие трамвайные линии;
- будущие автобусные маршруты;
- зоны риска;
- зоны роста;
- MPZP;
- planned investments.

Должны быть переключаемые слои.

### 6.3. История цены объекта

Для каждого объекта хранить:

- дату первого появления;
- стартовую цену;
- текущую цену;
- все изменения цены;
- цену за м² на каждый момент;
- количество дней на рынке;
- количество снижений цены;
- количество повышений цены;
- был ли объект снят;
- был ли объект опубликован повторно;
- менялись ли параметры объекта;
- менялось ли описание.

Система должна определять:

- объект давно висит;
- продавец снижал цену;
- цена выше рынка;
- цена ниже рынка;
- есть пространство для торга;
- объект может быть проблемным;
- объект может быть недооцененным.

### 6.4. Рыночная аналитика

Дашборды:

- средняя цена за м²;
- медианная цена за м²;
- динамика цены за м²;
- количество активных объявлений;
- новые объявления за период;
- снятые объявления за период;
- среднее время экспозиции;
- распределение цен;
- распределение по комнатам;
- распределение по площади;
- первичный / вторичный рынок;
- сравнение районов;
- сравнение городов;
- индекс перегретости района;
- индекс покупательского рынка;
- индекс продавцовского рынка;
- ликвидность района.

Примеры инсайтов:

- “В районе за 90 дней предложение выросло на 18%, а медианная цена снизилась на 4%. Это может говорить о слабом спросе.”
- “3-комнатных квартир в этом районе мало, а цена за м² растет быстрее среднего по городу.”
- “Объект на 9% дешевле медианы похожих квартир в радиусе 2 км.”
- “Похожие объекты продаются в среднем 74 дня, а этот висит 162 дня. Есть повод для торга.”

### 6.5. Анализ будущего развития района

Для каждого объекта анализировать радиусы:

- 500 м;
- 1 км;
- 2 км;
- 5 км;
- 10 км.

Факторы роста:

- новая трамвайная линия;
- новая автобусная линия;
- новая дорога;
- новая школа;
- новый детский сад;
- новый парк;
- новая больница;
- торговый центр;
- офисный центр;
- университет;
- улучшение транспорта;
- рост населения;
- рост рабочих мест;
- новые жилые проекты;
- благоустройство района.

Факторы риска:

- завод;
- промзона;
- складская зона;
- шумная трасса;
- железная дорога;
- аэропорт;
- риск затопления;
- загрязнение;
- слабый транспорт;
- отсутствие школ;
- переизбыток новостроек;
- плохая ликвидность;
- слишком быстрый рост цен;
- слабая арендная доходность.

### 6.6. Investment Score

Для каждого объекта рассчитывать рейтинг инвестиционной привлекательности от 0 до 100.

Пример факторов:

- цена относительно рынка;
- динамика цен района;
- ликвидность похожих объектов;
- транспортная доступность;
- будущие инвестиции;
- инфраструктура;
- арендный потенциал;
- демография;
- количество конкурирующих объектов;
- качество здания;
- этаж;
- планировка;
- состояние объекта;
- юридические риски;
- негативная инфраструктура;
- качество данных.

Пример шкалы:

- 80–100: высокий потенциал;
- 60–79: хороший вариант;
- 40–59: средний вариант;
- 0–39: слабый или рискованный вариант.

Важно: рейтинг не должен обещать рост цены. Нужно указывать, что это аналитическая оценка, а не финансовая рекомендация.

### 6.7. Risk Score

Отдельно от Investment Score нужен Risk Score.

Факторы риска:

- цена сильно выше рынка;
- объект долго продается;
- многократное снижение цены;
- промышленные объекты рядом;
- шум;
- плохой транспорт;
- плохая инфраструктура;
- старый фонд;
- последний/первый этаж;
- нет лифта;
- слабая ликвидность;
- сомнительное описание;
- неполные данные;
- высокая конкуренция в районе.

### 6.8. Negotiation Score

Показывать, можно ли торговаться.

Факторы:

- объект долго на рынке;
- цена снижалась;
- цена выше медианы;
- много похожих объектов рядом;
- объект публиковался повторно;
- продавец менял описание;
- район показывает слабый спрос.

Вывод:

- “низкий шанс торга”;
- “средний шанс торга”;
- “высокий шанс торга”;
- “есть аргументы для снижения цены”.

### 6.9. AI-помощник

AI должен отвечать на вопросы пользователя по объекту и району.

Примеры вопросов:

- “Стоит ли покупать эту квартиру?”
- “Цена нормальная?”
- “Можно ли торговаться?”
- “Какие риски?”
- “Что будет построено рядом?”
- “Подходит ли для семьи?”
- “Подходит ли для аренды?”
- “Почему объект дешевле рынка?”
- “Что проверить перед покупкой?”
- “Какие вопросы задать продавцу?”
- “Что лучше: объект A или объект B?”

AI должен отвечать только на основе данных платформы и указывать, какие данные использованы.

Запрещено:

- обещать гарантированный рост цены;
- давать юридические гарантии;
- давать финансовую рекомендацию без disclaimer;
- выдумывать данные, которых нет.

### 6.10. Сравнение объектов

Пользователь должен сравнивать 2–5 объектов по:

- цене;
- цене за м²;
- площади;
- комнатам;
- этажу;
- году постройки;
- району;
- транспорту;
- школам;
- инфраструктуре;
- будущим инвестициям;
- истории цены;
- Investment Score;
- Risk Score;
- Negotiation Score;
- ипотечному платежу;
- арендному потенциалу;
- ликвидности.

### 6.11. Новости недвижимости

Раздел новостей должен включать:

- новости рынка недвижимости Польши;
- новости по регионам;
- новости по Вроцлаву;
- изменения ипотечных программ;
- изменения налогов;
- изменения законодательства;
- новости девелоперов;
- городские инвестиции;
- транспортные изменения;
- решения по MPZP;
- аналитические обзоры.

Для каждой новости AI должен делать краткую сводку:

- что произошло;
- почему это важно;
- на какие районы влияет;
- как может повлиять на цены;
- кому это важно.

### 6.12. Ипотечный калькулятор

Функции:

- стоимость квартиры;
- первоначальный взнос;
- сумма кредита;
- срок кредита;

- общий бюджет покупки;
- сравнение сценариев;
- проверка доступности покупки по доходу.

## 7. MVP

MVP должен быть узким, но полезным.

### Регион MVP

- Wrocław;
- Dolnośląskie;
- радиус 20–30 км вокруг Вроцлава.

### Тип объектов

- квартиры на продажу.

### MVP-функции

1. Сбор объявлений из 1–2 источников.
2. Нормализация данных.
3. Дедупликация объектов.
4. История цены.
5. Поиск и фильтры.
6. Карта объектов.
7. Цена за м².
8. Сравнение объекта с похожими.
9. Базовая аналитика района.
10. Инфраструктура рядом.
11. Базовые слои будущих инвестиций.
12. Investment Score v1.
13. Risk Score v1.
14. Negotiation Score v1.
15. Карточка объекта.
16. Избранное.
17. Email/Telegram-уведомления.
18. PDF-отчет по объекту.
19. Базовый ипотечный калькулятор.
20. Страница тарифов.
21. Авторизация.
22. Админка для управления источниками и качеством данных.

### Что не делать в MVP

Не делать сразу:

## 9. Скоринг и аналитика

### 9.1. Investment Score

Пример весов:

- цена относительно рынка: 20%;
- динамика района: 15%;
- транспортная доступность: 15%;
- будущие инвестиции: 15%;
- ликвидность: 10%;
- инфраструктура для жизни: 10%;
- арендный потенциал: 10%;
- качество данных: 5%;
- риски: отрицательный коэффициент.

### 9.2. Risk Score

Факторы:

- промзоны;
- шум;
- плохой транспорт;
- объект долго продается;
- цена выше рынка;
- плохая инфраструктура;
- низкая ликвидность;
- неполные данные;
- переизбыток предложения.

### 9.3. Liquidity Score

Факторы:

- среднее время продажи похожих объектов;
- количество похожих объектов;
- динамика спроса;
- популярность района;
- количество снижений цены;
- активность рынка.

### 9.4. Rental Potential Score

Факторы:

- близость транспорта;
- близость университетов;
- близость офисов;
- спрос на аренду;
- цена покупки;
- примерная арендная ставка;
- конкуренция;
- район.

### 9.5. Fair Price Estimate

Система должна оценивать справедливую цену объекта на основе:

- похожих объектов;
- района;
- площади;
- комнат;
- этажа;
- года постройки;
- состояния;
- инфраструктуры;
- истории рынка;
- цены за м².

## 12. Go-to-Market Strategy

### 12.1. Первый сегмент

Начать не с массового B2C, а с B2B/B2Pro:

- риелторы;
- маленькие агентства;
- инвесторы;
- ипотечные брокеры.

Почему:

- они платят регулярно;
- им нужны отчеты;
- они понимают ценность аналитики;
- они могут дать обратную связь;
- они могут привести первых клиентов.

### 12.2. Быстрая проверка спроса

Сделать landing page:

- “Проверка квартиры перед покупкой”;
- “PDF-отчет по объекту и району”;
- “История цены и аргументы для торга”;
- “Аналитика для риелторов”.

Запустить:

- Facebook группы по недвижимости;
- группы украинцев/белорусов/иностранцев в Польше;
- LinkedIn для риелторов;
- холодные сообщения агентствам;
- Google Ads по запросам “czy warto kupić mieszkanie Wrocław”, “cena za m2 Wrocław”, “analiza mieszkania przed zakupem”;
- SEO-страницы по районам.

### 12.3. Первый платный продукт

Не ждать полного SaaS.

Сначала продавать вручную или полуавтоматически:

- отчет по квартире;
- отчет по району;
- отчет для риелтора;
- подборку объектов ниже рынка.

Это позволит проверить, готовы ли платить.
